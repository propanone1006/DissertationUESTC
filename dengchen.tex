	% 开启中文摘要
	\zhabstract
	
	随着大语言模型（LLM）在“模型即服务（MaaS）”云架构下的广泛部署，推理服务的效率与安全性成为了学术界与工业界关注的焦点。为打破自回归生成的算力与显存瓶颈，现代高性能推理引擎（如 vLLM）引入了 PagedAttention 与前缀缓存（Prefix Caching）机制。然而，这些底层资源调度机制在提升系统吞吐量的同时，也暴露出严重的新型安全威胁。传统的网络层防火墙与基于语义分析的检测方法，在面对利用合法 API 实施的底层资源消耗型攻击时往往完全失效。针对这一问题，本文深入研究了基于多维资源特征的大语言模型恶意请求检测方法，主要研究工作与成果如下：

	首先，本文剖析了大模型双阶段推理机制中的资源消耗异构性，揭示了预填充（Prefill）阶段的算力瓶颈与解码（Decode）阶段的访存瓶颈。在此基础上，明确了两种隐蔽的威胁模型：利用高并发短文本榨干 GPU 算力的拒绝服务（DDoS）攻击，以及利用均匀分布请求引发缓存频繁置换的共享缓存侧信道（Side-Channel）攻击。

	其次，针对上述攻击导致的时间序列畸变，本文提出了一种基于滑动时间窗口的多维特征提取方法。深入挖掘了首 Token 延迟（TTFT）在攻击下的“双峰分离”与长尾现象，并引入峰度（Kurtosis）等高阶统计量，结合每 Token 平均延迟（TPOT）、系统队列长度及缓存命中率，构建了刻画底层物理资源状态的多维特征工程体系。

	最后，本文设计并实现了一种轻量级的两阶段异常检测框架。针对云端系统由于网络波动产生的分布外（OOD）极值噪声，引入孤立森林（Isolation Forest）算法进行前置数据清洗；针对清洗后的高质量特征流，构建基于单边梯度采样与直方图优化的 LightGBM 多分类模型。实验结果表明，该检测框架能够精准识别并分类正常流量、DDoS 攻击与侧信道缓存污染攻击，且模型自身的推理开销极低，在保障大模型服务高可用性的同时，为云端推理平台提供了实时、高效的安全防护。

	\enabstract
	
	With the widespread deployment of Large Language Models (LLMs) in Model-as-a-Service (MaaS) cloud architectures, the efficiency and security of inference serving have become focal points for both academia and industry. To overcome the computational and memory bottlenecks of autoregressive generation, modern high-performance inference engines (such as vLLM) have introduced mechanisms like PagedAttention and Prefix Caching. However, while enhancing system throughput, these underlying resource scheduling mechanisms inadvertently expose critical new security vulnerabilities. Traditional network-layer firewalls and semantic-based detection methods often fail completely when facing resource-exhaustion attacks executed via legitimate APIs. To address this issue, this thesis deeply investigates a malicious request detection method for LLMs based on multi-dimensional resource features. The main research work and contributions are as follows:

	First, this thesis analyzes the heterogeneity of resource consumption in the two-stage LLM inference mechanism, revealing the compute-bound nature of the prefill stage and the memory-bound nature of the decode stage. Based on this, two stealthy threat models are identified: Distributed Denial of Service (DDoS) attacks that exhaust GPU computational power using high-concurrency short texts, and shared cache side-channel attacks that trigger frequent cache evictions using uniformly distributed requests.

	Second, in response to the time-series distortions caused by the aforementioned attacks, a multi-dimensional feature extraction method based on sliding time windows is proposed. This study thoroughly mines the "twin-peak separation" and heavy-tail phenomena of the Time to First Token (TTFT) under attack. By introducing high-order statistics such as Kurtosis, combined with Time Per Output Token (TPOT), system queue length, and cache hit rates, a multi-dimensional feature engineering system is constructed to characterize the underlying physical resource states.

	Finally, this thesis designs and implements a lightweight, two-stage anomaly detection framework. To handle extreme Out-of-Distribution (OOD) noise caused by network fluctuations in cloud systems, the Isolation Forest algorithm is introduced for preliminary data cleaning. For the cleaned high-quality feature stream, a LightGBM multi-classification model optimized with Gradient-based One-Side Sampling and histogram algorithms is constructed. Experimental results demonstrate that the proposed detection framework can accurately identify and classify normal traffic, DDoS attacks, and side-channel cache pollution attacks. With minimal inference overhead, the model ensures the high availability of LLM services and provides real-time, efficient security protection for cloud-based inference platforms.

	\chapter{绪论}
	\section{研究背景与意义}
	% LLM 推理服务在架构形态与资源使用模式上呈现出显著差异。

	% LLM 推理请求在执行过程中会触发 token 级别的逐步生成，伴随显存占用、KV 缓存分配与多阶段计算（prefill 与 decode）等复杂行为，其资源消耗不仅与请求频率相关，还高度依赖于输入提示（prompt）结构、上下文长度以及生成策略参数。

	% 多租户云环境下，来自不同用户和业务的高并发请求会在共享 GPU、显存和缓存资源的条件下被统一调度，推理负载之间的相互干扰和资源竞争显著增强。
	
	随着深度学习技术的突破，以 Transformer 为核心架构的大语言模型（LLM）在自然语言处理领域展现出了前所未有的理解与生成能力，	标志着人工智能正式步入生成式时代。在这种技术浪潮下，企业与开发者日益倾向于采用“模型即服务”（Model-as-a-Service, MaaS）的部署模式，将庞大的模型托管于云端高性能算力集群中，通过标准化的 API 对外提供推理能力。这种高度集约化的云服务模式虽然极大地降低了用户端的使用门槛，但也使得云端推理系统的稳定性和安全性成为了支撑大规模 AI 应用的关键基石。

	然而，大语言模型的推理过程与传统的 Web 服务或计算任务存在本质差异，其表现出显著的计算密集型与内存敏感型特征。为了应对海量用户并发请求并降低推理延迟，现代推理框架如 vLLM 引入了 PagedAttention 与 Prefix Caching 等先进的内存管理技术。这些技术通过对键值缓存（KV Cache）的精细化调度，虽然显著提升了吞吐量，但也无意中暴露了新的安全边界。攻击者可以利用推理系统对资源消耗的高度敏感性，通过构造特定模式的输入请求，诱发系统出现资源耗尽或性能剧烈抖动，这种针对算法逻辑与系统架构层面的攻击手段比传统的网络层攻击更具隐蔽性。

	在现有的网络安全防御体系中，传统的防火墙、入侵检测系统（IDS）以及 Web 应用防火墙（WAF）主要聚焦于网络协议层的特征匹配或已知恶意代码的拦截。对于大模型推理服务而言，攻击请求在格式上通常表现为完全合法的自然语言指令，但在推理执行阶段却会引发异常的显存占用或超长计算时间。例如，攻击者可能利用 Uniform 分布的请求模式破坏缓存的局部性原理，从而实施缓存侧信道干扰；或者通过高并发短文本请求制造拒绝服务攻击，压垮调度系统。这种“应用逻辑层”的资源压榨行为使得现有的基于流量签名的防御手段难以奏效，迫切需要研究一种能够深度感知模型推理行为、捕捉底层资源波动特征的新型检测机制。

	本研究的开展在学术理论维度具有重要意义，其通过对大模型推理过程中的多维性能指标进行精细化建模，填补了生成式 AI 基础设施安全防御的研究空白。研究不再局限于宏观的吞吐量统计，而是深入探讨了首 Token 延迟（TTFT）、每 Token 平均延迟（TPOT）以及 KV Cache 命中率等微观指标与攻击行为之间的内在耦合关系。通过对比Qwen 的不同参数规模模型在受攻击状态下的特征表现，本研究揭示了攻击流量在系统层留下的“特征足迹”，为构建大模型环境下异常行为的特征工程提供了科学的方法论指导，有助于完善人工智能安全保障体系。

	在实际应用价值方面，本研究所提出的攻击检测方案能够为云服务提供商提供坚实的安全防护屏障。在昂贵的 GPU 算力资源面前，任何形式的资源耗尽攻击都会直接导致服务商运营成本的激增以及服务等级协议（SLA）的违约。有效的检测系统能够精准识别旨在干扰 KV Cache 机制的隐蔽攻击，在保障正常用户请求获得低延迟响应的同时，极大程度地提高了算力资源的有效利用率。通过实时监控并阻断针对推理框架漏洞的恶意嗅探与冲击，可以有效防止系统雪崩效应的发生，确保大规模 AI 推理服务的可用性与经济性。

	此外，针对缓存侧信道等涉及多租户环境隐私与公平性的新型威胁，本研究提出的检测方法具备极强的实战防御意义。侧信道攻击往往通过探测缓存池状态的细微波动来推断其他用户的访问模式或干扰系统公平调度，这类攻击具有极高的技术门槛和防御难度。本研究通过对显存迁移频次、缓存失效分布等关键指标的异常分析，能够实现对此类高级持续性威胁（APT）的早期预警与快速响应。这不仅为构建可信人工智能环境提供了关键技术支持，也为生成式人工智能在金融、政务等对安全性要求严苛领域的深层次落地应用扫清了障碍。
	
	\section{国内外研究现状}

	为有效应对大模型推理服务面临的安全威胁，国内外学术界和工业界已经展开了多维度的研究探索。通过阅读大量相关文献，本文对目前常见的恶意请求检测与防御方法进行了总结归纳，主要将其分为三类：基于输入内容的攻击检测、基于系统行为与微观架构的攻击检测，以及基于分布外检测（OOD）与机器学习的异常识别方法。本小节将依次进行介绍。 

	\subsection{基于输入内容的攻击检测}
	基于输入内容的攻击检测是目前大语言模型安全领域最主流、应用最广的研究赛道，主要针对提示词注入（Prompt Injection）、越狱攻击（Jailbreak）以及对抗样本攻击\cite{检测NLP中的对抗样本}。此类方法的核心思想是通过对用户输入的自然语言指令进行语义分析、特征提取或困惑度（Perplexity, PPL）计算，从而拦截恶意请求。

	在对抗攻击防御方面，Jain 等人对齐语言模型面临的对抗样本进行了系统性研究，并证明了基于困惑度（PPL）的启发式过滤器能够作为强基线防御手段，有效检测并拦截某些由乱码或异常词汇构成的对抗攻击\cite{基线防御针对对齐语言模型的对抗攻击}。Armstrong 等人也深入探讨了在自然语言处理中检测对抗样本的多种特征工程方法\cite{检测NLP中的对抗样本}。然而，这类基于语义和内容特征的检测方法存在一个致命的盲区：当攻击者旨在发起资源耗尽型 DDoS 攻击或缓存侧信道攻击时，其构造的提示词（Prompt）在语义上往往是完全合法的正常语句（例如我们在压测中使用的常规长文本或随机短文本）。此时，传统的输入内容过滤机制将完全失效，恶意请求会畅通无阻地进入推理计算底座。

	\subsection{基于系统行为与微观架构的攻击检测}
	随着大模型云服务（MaaS）的普及，攻击者的目标逐渐从“欺骗模型生成有害内容”转向“压榨底层算力与窃取隐私”。因此，基于系统行为和微观架构（尤其是针对 KV Cache 和调度机制）的攻击与检测成为了当前安全研究的最前沿。

	在针对大模型推理接口的定时侧信道（Timing Side-Channel）研究方面，Carlini 等人率先证明了可以通过远程定时攻击来推断高效语言模型的内部状态\cite{高效推理接口上的远程定时攻击}。随后，Song 等人进一步揭示了大型语言模型服务系统（如 vLLM）中由于批处理（Batching）和缓存机制引入的定时侧信道漏洞\cite{早起的鸟儿发现漏洞：揭示大型语言模型服务系统中的定时侧信道}。Zhang 等人则从输出层入手，展示了如何通过输出 Token 数量的定时波动来建立侧信道\cite{输出Token数量的定时侧信道攻击}。

	针对云端共享 KV Cache 带来的多租户隔离问题，国内外学者进行了深入剖析。Zheng 等人提出 InputSnatch 攻击，证明了在共享大模型服务中，攻击者可以通过定时侧信道窃取其他用户的输入信息\cite{共享缓存中的定时侧信道攻击}。Luo 等人也发文揭示了 KV Cache 在大模型推断中隐藏的严重隐私风险，并初步探讨了缓解机制\cite{缓存中的阴影：揭示并缓解KV-缓存在大型语言模型推断中的隐私风险}。此外，McDonald 等人提出的 Whisper Leak 端侧信道攻击，更是通过系统资源消耗特征的细微变化，实现了对 LLM 内部计算过程的窥探\cite{对大型语言模型的端侧信道攻击}。

	在宏观的云端拒绝服务（DDoS）检测方面，Deniz 和 Serttaş 针对云网络架构，构建了基于卷积神经网络（CNN）和长短期记忆网络（LSTM）的深度学习分布式拒绝服务检测系统，并取得了高达 98\% 的准确率\cite{基于深度学习的云网络中分布式拒绝服务检测系统}。然而，传统的云原生 DDoS 检测多基于网络连接数或带宽阈值，尚未充分结合大模型预填充（Prefill）与解码（Decode）双阶段的非线性算力消耗特征。

	\subsection{基于机器学习与分布外检测（OOD）的防御方法}
	鉴于大模型推理过程中的攻击流量（如并发压榨或 Uniform 分布的缓存污染）往往在首 Token 延迟（TTFT）和每 Token 平均延迟（TPOT）等时间序列指标上表现出异常波动，基于机器学习（ML）与分布外检测（Out-of-Distribution, OOD）的技术成为了构建防御体系的重要手段。 

	在传统的硬件与缓存侧信道检测中，Tong 等人已经证明了基于机器学习分类器能够高精度地识别缓存侧信道攻击的微架构足迹\cite{基于机器学习的缓存侧信道攻击检测}。针对多维时间序列的异常检测问题，Malhotra 等人提出了基于 LSTM 的编码器-解码器架构，通过学习正常系统状态的重构误差，有效识别多传感器环境下的分布外异常点\cite{基于LSTM的编码器-解码器用于多传感器异常检测}。此外，Zhang 等人提出的 CloudRadar 也展示了深度学习在实时云环境侧信道检测中的潜力\cite{云网络中基于深度学习的分布式拒绝服务检测系统}。
	
	这些研究为大模型环境下的异常流量识别奠定了理论基础。然而，在云原生大模型推理场景下，面对极高的低延迟（Low-latency）与高并发吞吐要求，过于沉重的深度神经网络（如前文所述的 LSTM、CNN）在旁路部署时，极易成为系统算力底座的额外负担（Overhead）。同时，正如学术界在探索黑盒模型时所面临的共性挑战，纯粹的深度学习预测往往缺乏对底层 Waiting 队列与物理显存调度（PagedAttention）机制的可解释性。

	综上所述，尽管目前针对 NLP 语义层的防御和基于深度学习的传统云安全检测方法已相对成熟，但面向大模型推理引擎底层的资源压榨（DDoS）与缓存干扰（KV Cache Side-Channel），依然缺乏一种兼顾“低推断延迟”与“高物理可解释性”的检测机制。这正是本文引入 TTFT 双峰分离与多维队列特征，并最终摒弃庞大的深度学习架构，转而选择具有极低延迟开销的树模型（如 LightGBM）开展异常检测研究的核心动机与切入点。


	\section{论文主要工作及创新}

	本文针对云服务环境下大语言模型（LLM）推理服务面临的资源安全威胁，构建了一套全流程的异常检测与分析体系。研究立足于高性能推理框架 vLLM，通过深度解析其 PagedAttention 内存管理机制与 Prefix Caching 缓存策略，剖析了系统在应对非典型流量冲击时的脆弱性本质。
	为实现对推理行为的精准表征，本文设计并实现了一套多维指标监控系统，能够实时采集首 Token 延迟（TTFT）、每 Token 平均延迟（TPOT）及 KV Cache 命中率等高保真底层特征，并在此基础上开展了如下针对性研究：

	（一）针对大模型推理过程中“合法指令”掩盖“恶意负载”导致的检测难问题。
	本文提出了基于“双峰分离”物理特征的推理攻击识别方法。在实验验证阶段，通过在 A100 环境下对比 Qwen1.5-1.8B 与 Qwen2.5-7B 模型发现，由于小模型计算延迟极低导致正常与攻击特征高度重叠，而随着模型参数规模提升至 7B，正常请求与高并发 DDoS 攻击请求在首 Token 延迟（TTFT）上的分布会演变为清晰的双峰分离态势。本文利用这一规模敏感型特征，突破了传统网络层检测无法感知模型内部计算状态的局限，显著提升了在复杂并发环境下识别资源压榨型攻击的鲁棒性。

	（二）针对利用 LLM 缓存逻辑缺陷实施的隐蔽侧信道攻击问题。
	本文创新性地构建了深度关联推理引擎内存调度层的检测机制。不同于侧重文本内容的审计手段，本文重点关注攻击者通过构造符合 Uniform 分布的随机请求来破坏缓存局部性的行为。研究揭示了攻击请求如何通过诱导 KV Cache 命中率从高位（Zipf 分布状态）剧烈下滑，并引发显存频繁迁移，进而造成系统整体吞吐性能退化。通过引入孤立森林等机器学习算法，本文实现了从请求分布偏移视角对隐蔽性侧信道干扰的高效捕捉，为防御大模型特有机制的逻辑攻击提供了全新观测维度。

	（三）针对安全检测系统在高并发环境下对推理性能产生侵入式干扰的问题。
	本文实现了一套具有高可扩展性的轻量化监控与检测框架。该框架能够无缝集成于生产级推理引擎 vLLM 之中，通过多维特征关联分析方法，将复杂的底层资源波动转化为直观的攻击判定指标。实验评价表明，该方案在保障高准确率与召回率的同时，极大地降低了监控模块对 GPU 推理性能的额外开销。这一工程实践为云端大模型服务的安全基座建设提供了具备实际应用价值的解决方案，确保了安全防御与服务效能的平衡。

	\section{大语言模型推理机制与算力瓶颈}
	\label{sec: 大语言模型推理机制与算力瓶颈}

	\subsection{Transformer 架构与自回归解码}
	目前主流的大语言模型（Large Language Models, LLM），如 GPT 系列、LLaMA 以及本文实验所采用的 Qwen 系列，均采用仅解码器（Decoder-only）的 Transformer 架构\cite{注意力机制是所有你需要的一切}。其核心特征是自回归生成（Autoregressive Generation），即模型在给定上文的条件下，通过概率分布逐个预测序列中的下一个标记（Token）。
	
	假设输入序列为 $\mathbf{X}_{1:n} = \{x_1, x_2, \dots, x_n\}$，其中 $x_i \in \mathbb{R}^{d_{model}}$ 为经过词表嵌入（Embedding）后的特征向量。在生成第 $n+1$ 个 Token 时，模型的目标是最大化条件概率：
	\begin{equation}	
		P(x_{n+1} \mid \mathbf{X}_{1:n}) = \text{Softmax}\left(\mathbf{W}_u \cdot \text{DecoderBlock}(\mathbf{X}_{1:n})\right)
	\end{equation}
	其中，$\mathbf{W}_u$ 为最终的输出词表映射矩阵（Unembedding Matrix）。

	在 Transformer 的每个解码器层中，最核心的计算单元为掩码多头自注意力机制（Masked Multi-Head Self-Attention）。对于输入矩阵 $\mathbf{X} \in \mathbb{R}^{L \times d_{model}}$（$L$ 为序列长度），系统首先通过三个可学习的线性权重矩阵 $\mathbf{W}_Q, \mathbf{W}_K, \mathbf{W}_V \in \mathbb{R}^{d_{model} \times d_k}$ 将其投影为查询矩阵（Query）、键矩阵（Key）和值矩阵（Value）：
	\begin{equation}
		\mathbf{Q} = \mathbf{X}\mathbf{W}_Q, \quad \mathbf{K} = \mathbf{X}\mathbf{W}_K, \quad \mathbf{V} = \mathbf{X}\mathbf{W}_V
	\end{equation}
	随后，通过缩放点积注意力计算当前 Token 与历史 Token 的关联权重：
	\begin{equation}
		\mathrm{Attention}(\mathbf{Q},\mathbf{K},\mathbf{V}) = \mathrm{Softmax}\left(\frac{\mathbf{Q}\mathbf{K}^T}{\sqrt{d_k}}\right)\mathbf{V}
	\end{equation}
	其中，$d_k$ 为注意力头的维度。在朴素的自回归生成中，由于序列长度 $L$ 会随着生成的进行不断增加，第 $t$ 步的计算不仅需要处理新生成的 Token，还需要重新计算前 $t-1$ 个历史 Token 的 $\mathbf{K}$ 和 $\mathbf{V}$ 矩阵，导致计算复杂度呈 $\mathcal{O}(L^2)$ 的二次方级增长，造成了极大的算力冗余。

	\subsection{KV Cache 加速原理与内存映射}
	为了消除上述 $\mathcal{O}(L^2)$ 复杂度的冗余特征提取计算，现代 LLM 推理框架普遍引入了键值缓存（KV Cache）机制。其核心思想是“以空间换时间”：在逐词生成的过程中，将历史 Token 计算得到的键向量 $k$ 和值向量 $v$ 缓存在 GPU 的物理显存（HBM）中，供后续生成步直接复用。

	如图\ref{fig: 第3步推理有无KVC加速对比示意图}所示，以生成第 3 个 Token 为例，本节对有无 KV Cache 机制的内部计算流程进行了对比说明。

		\caption{第3步推理有无KVC加速对比示意图} \label{fig: 第3步推理有无KVC加速对比示意图}
	\end{figure}

	在无 KV Cache 的全量计算模式下（图左侧），系统必须重新将前两个 Token 与当前 Token 拼接，执行完整的矩阵乘法（GEMM）运算。而在引入 KV Cache 的增量计算模式下（图右侧），设当前为第 $t$ 步，此时显存中已经保存了前 $t-1$ 步的缓存矩阵 $\mathbf{K}_{\text{cache}}^{(t-1)}$ 与 $\mathbf{V}_{\text{cache}}^{(t-1)}$。模型仅需计算当前最新 Token $x_t$ 对应的查询向量 $q_t \in \mathbb{R}^{1 \times d_k}$、键向量 $k_t$ 与值向量 $v_t$。
	随后，系统将新的 $k_t, v_t$ 向量与历史缓存沿序列长度维度进行张量拼接（Concatenation）：
	\begin{equation}
		\mathbf{K}_{\text{cache}}^{(t)} = \left[ \mathbf{K}_{\text{cache}}^{(t-1)} \oplus k_t \right], \quad
		\mathbf{V}_{\text{cache}}^{(t)} = \left[ \mathbf{V}_{\text{cache}}^{(t-1)} \oplus v_t \right]
	\end{equation}
	此时，第 $t$ 步的注意力计算被大幅简化为一次矩阵-向量乘法（GEMV）：
	\begin{equation}
		\text{Attention}(q_t, \mathbf{K}_{\text{cache}}^{(t)}, \mathbf{V}_{\text{cache}}^{(t)}) = \text{Softmax} 
		\left( \frac{q_t (\mathbf{K}_{\text{cache}}^{(t)})^T}{\sqrt{d_k}} 
		\right) \mathbf{V}_{\text{cache}}^{(t)}
	\end{equation}
	如图\ref{fig: 第3步推理有无KVC加速对比示意图}中的灰色区块所示，历史特征被直接从显存中读取，极大地提升了推理速度。然而，这种机制也导致显存的占用量随着并发请求数和生成长度的增加而线性膨胀，使 KV Cache 成为大模型推理系统中最核心也最脆弱的资源池。

	\subsection{双阶段推理的算力瓶颈分析：Prefill 与 Decode}
	在引入 KV Cache 后，大语言模型的单次推理生命周期在物理上被明确划分为具有截然不同资源消耗特征的阶段：预填充阶段（Prefill Phase）与解码阶段（Decoding Phase）。这种底层物理特征的割裂，正是本文后续进行多维异常特征提取与攻击检测的理论根基。

	为了更直观地阐述这种物理层面的割裂，本文绘制了大模型双阶段推理机制与底层资源消耗的对比图。如图\ref{fig: 大语言模型双阶段推理机制与资源消耗对比图}所示，本文将大模型单次推理的生命周期及其底层硬件状态进行了可视化分解。

	\textbf{（1）预填充阶段（Prefill Phase）与计算墙（Compute-bound）}

	当用户请求首次到达系统时，由于此时 KV Cache 为空，模型需要并行处理用户输入的全部提示词（Prompt）序列（长度为 $L$）。这是一个典型的大规模矩阵-矩阵乘法（GEMM）过程。现代 GPU（如 NVIDIA A100）内部的 Tensor Core 在执行此类高并发并行运算时效率极高，因此该阶段的瓶颈在于算力（FLOPs），而非内存带宽，被称为计算密集型（Compute-bound）任务。用户感知到的首 Token 延迟（Time to First Token, TTFT）主要由该阶段的计算耗时以及在系统调度队列中的等待耗时决定。在面对本文第三章所述的 DDoS 攻击时，高并发的短文本请求会瞬间榨干 GPU 的算力资源，导致 TTFT 剧烈飙升。

	\textbf{（2）解码阶段（Decoding Phase）与内存墙（Memory-bound）} 

	进入逐词生成阶段后，由于每次只输入一个新的 Token，注意力计算退化为矩阵-向量乘法（GEMV）。尽管算力需求骤降，但为了计算这一个 Token，GPU 必须从高带宽显存（HBM）中将庞大的历史 KV Cache 数据完整地搬运到静态随机存取存储器（SRAM）中。此时，系统的性能瓶颈转移至显存带宽，成为访存密集型（Memory-bound）任务，决定了每 Token 平均延迟（Time Per Output Token, TPOT）。当遭受本文第四章所述的缓存侧信道攻击时，由于攻击者构造的 Uniform 分布请求不断触发 Cache Miss，迫使系统频繁在 Prefill 和 Decode 状态间切换并进行低效的显存页调度，从而在 TTFT 和 TPOT 上留下明显的畸变特征。

	\section{高性能推理系统的内存调度机制}
    \label{sec:高性能推理系统的内存调度机制}
    
    在明确了大模型推理双阶段的算力与显存瓶颈后，如何高效管理底层资源成为了提升系统并发处理能力（CONC）的关键所在。在提高大语言模型推理吞吐量的工程实践中，除了本文重点研究的 vLLM 的 PagedAttention 机制外，学术界与工业界也提出了诸多底层的优化方案。例如，Dao 等人提出的 FlashAttention 机制通过硬件 I/O 感知极大地加速了精确注意力机制的计算效率并降低了显存读写开销\cite{FlashAttention快速且内存高效的精确注意力机制}。同时，微软提出的 DeepSpeed-Inference 框架则为前所未有规模的 Transformer 模型提供了极其高效的分布式推理能力\cite{DeepSpeed实现前所未有规模的Transformer模型高效推理}。这些底层计算加速技术共同构成了当前云原生 MaaS 平台的算力基石。
    
    聚焦回显存调度本身，本文实验所采用的 vLLM 引擎，通过引入操作系统级别的虚拟内存管理思想，从根本上重构了 KV Cache 的分配机制。本节将深入剖析其核心的 PagedAttention 技术与 Prefix Caching 策略。

	\subsection{传统内存管理的碎片化困境}
	在传统的深度学习推理框架（如早期的 FasterTransformer 或 HuggingFace Accelerate）中，系统通常采用连续显存分配策略。由于大语言模型在生成初期无法预知最终输出的 Token 数量，系统只能根据模型支持的最大序列长度（Max Sequence Length）为每个请求预先静态分配一块庞大且连续的 GPU 显存空间。

	这种“最坏情况假设”导致了极其严重的内存浪费。具体表现为：
	\begin{itemize}
		\item \textbf{内部碎片（Internal Fragmentation）}：当实际生成的序列远短于预留长度时，剩余的预留显存被长期闲置且无法被其他请求使用。
		\item \textbf{外部碎片（External Fragmentation）}：由于不同请求释放显存的时间不一，物理显存中会产生大量不连续的空闲片段。当新的大内存需求到来时，即使总空闲容量足够，也因缺乏连续空间而触发内存溢出（Out of Memory, OOM）。
	\end{itemize}
	在高并发的多租户云服务环境中，这种低效的内存管理模式极大地限制了系统吞吐量，也使得系统在面对海量突发请求（如拒绝服务攻击）时极为脆弱。

	\subsection{PagedAttention 原理与块表映射}
	为了打破连续内存分配的桎梏，vLLM 提出了 PagedAttention 算法\cite{利用PagedAttention进行大型语言模型服务的高效内存管理}。该算法借鉴了现代操作系统中的虚拟内存与分页（Paging）机制，将原本必须连续存储的 KV Cache 划分为固定大小的逻辑块（Logical Blocks），并将其动态映射到非连续的物理显存块（Physical Blocks）上。

	在 PagedAttention 中，每个物理块能够容纳固定数量的 Token（例如在本文实验配置中，默认块大小为 16 个 Token）。系统通过维护一张全局的块表（Block Table），记录每个请求的逻辑块到物理块的映射关系：
	\begin{equation}
		\text{BlockTable}(\text{Req}_i) = \{pb_1, pb_2, \dots, pb_m\}
	\end{equation}
	其中，$\text{Req}_i$ 为第 $i$ 个请求，$pb$ 为分配的物理块索引。当模型在解码阶段生成新的 Token 时，系统只需按需分配新的物理块并更新块表，而无需进行连续的内存预留。

	\begin{figure}[!htb]
		\centering
		\includegraphics[width=0.95\linewidth]{dc_PagedAttention 逻辑块映射与 Prefix Caching 共享复用机制} % 替换为你的实际图片文件名
		\caption{PagedAttention 逻辑块映射与 Prefix Caching 共享复用机制} \label{fig: PagedAttention 逻辑块映射与 Prefix Caching 共享复用机制}
	\end{figure}

	如图\ref{fig: PagedAttention 逻辑块映射与 Prefix Caching 共享复用机制}所示，在 PagedAttention 机制的调度下，连续的逻辑序列被切分为大小固定的逻辑块（Logical Blocks），并通过块表（Block Table）离散地映射到 GPU 物理显存中的非连续物理块（Physical Blocks）上，从而彻底消除了传统内存分配中的内部碎片。

	在此基础上，图例进一步展示了 Prefix Caching 的前缀复用工作原理：假设请求 A（Req A）与请求 B（Req B）在输入时包含了相同的系统前缀（如相同的人设指令）。vLLM 引擎会计算该前缀逻辑块的哈希值，并发现物理显存中已存在对应的缓存块。此时，系统不再为请求 B 分配新的物理空间，而是将其对应的逻辑块指针直接映射至请求 A 已生成的物理块上，并同步更新该物理块的引用计数（Reference Count）。这种内存页级别的细粒度共享与按需分配机制，在极大提升系统并发上限（即本文实验中的 CONC 指标）的同时，也使得物理显存池成为了多租户之间相互影响的共享媒介，为后续的侧信道干扰埋下了伏笔。

	通过这种页式内存管理，PagedAttention 彻底消除了内部碎片，并将外部碎片率控制在极低水平（仅最后一个未填满的块存在微小浪费），从而允许系统在同一块 A100 GPU 上承载远超传统框架的并发请求数（即本文压测实验中极高的 CONC 值）。然而，复杂的逻辑地址映射也为系统底层的调度器带来了额外的算力开销。

	\subsection{Prefix Caching 与前缀复用漏洞}
	在实际的“模型即服务（MaaS）”场景中，大量的用户请求往往包含相同的系统提示词（System Prompt）、任务指令或长背景文档。为了进一步压榨显存价值，vLLM 在 PagedAttention 的基础上引入了 Prefix Caching（前缀缓存）机制。

	Prefix Caching 的核心逻辑是实现多租户间的物理块共享。当新请求到达时，系统会对其输入的 Prompt 按逻辑块进行哈希（Hash）计算：
	\begin{equation}
		\text{Hash}_{block} = \mathcal{H}(\text{Token}_1, \dots, \text{Token}_k)
	\end{equation}
	系统在全局哈希表中检索该哈希值。若发生缓存命中（Cache Hit），则不再为该请求分配新的物理块，而是直接将该请求的逻辑块指针指向已存在于物理显存中的共享块，并增加该物理块的引用计数（Reference Count）。

	\textbf{正常长尾分布与缓存局部性}：在正常的业务流量中，用户请求通常符合 Zipf 分布（长尾分布），少数热点提示词被高频复用，此时 Prefix Caching 能维持极高的命中率，显著降低 TTFT 并提升吞吐量。

	\textbf{新型安全威胁面的暴露}：然而，这种基于多租户共享的缓存机制也无意中暴露了侧信道安全边界。当攻击者恶意构造符合 Uniform 分布（完全随机分布）的大量请求时，不仅无法命中任何缓存，还会迫使系统频繁为新的“冷前缀”分配物理块。当物理显存耗尽时，系统将依据置换算法（如 LRU）无差别地驱逐（Evict）正常用户的热点缓存。这种利用合法 API 实施的“缓存污染（Cache Pollution）”攻击，能够在不引起网络层流量异常的情况下，精准破坏大模型推理的底层局部性原理。这也构成了本文第四章基于缓存侧信道进行异常检测的物理与逻辑基础。

	\section{针对大模型推理服务的新型安全威胁模型}
	
	在深入理解 vLLM 的双阶段算力瓶颈与 PagedAttention 内存调度机制后，本节将正式定义本文研究的安全威胁模型（Threat Model）。与传统的网络层拒绝服务攻击不同，针对大语言模型（LLM）的攻击具有“应用层合法、物理层致命”的隐蔽特征。

	\subsection{攻击者能力与假设}
	在本文的威胁模型中，假设云服务商采用“模型即服务（MaaS）”架构对外提供 API 接口。攻击者（Attacker）具备以下能力与限制：
	\begin{itemize}
		\item \textbf{黑盒访问权限}：攻击者无法直接获取服务器的物理控制权，无法修改模型权重，也无法读取宿主机的底层操作系统日志。其唯一的交互途径是通过公开的 API 端点发送自然语言提示词（Prompt）并接收生成的 Token。
		\item \textbf{合法身份掩护}：攻击者拥有合法的注册账号与访问令牌（Token ID），其发送的请求内容在语义上完全符合应用层安全防火墙（WAF）的过滤规则，不存在 SQL 注入或传统越狱（Jailbreak）等违规字符串。
		\item \textbf{流量控制能力}：攻击者能够利用自动化脚本并发发送请求，并能自由控制输入 Prompt 的长度、并发数（CONC）以及请求内容的分布概率（如调整 Zipf 分布的 $\alpha$ 参数）。
	\end{itemize}

	\subsection{拒绝服务（DDoS）攻击面：算力墙压榨}
	结合 \ref{sec: 大语言模型推理机制与算力瓶颈} 节的双阶段理论，预填充（Prefill）阶段是极度消耗 GPU Tensor Core 算力的矩阵乘法过程。攻击者利用这一非线性算力消耗特征，实施大模型特有的 DDoS 攻击。

	攻击者通过脚本在极短时间内发送海量（如并发数超过系统最大承载量）的“短文本”请求。由于短文本在网络传输层面占用的带宽极小，极易穿透传统的网络层流量清洗设备。然而，当这些请求涌入 vLLM 的调度器时，系统必须为每一个请求启动耗时的 Prefill 计算。瞬间爆发的算力需求会迅速榨干 GPU 的计算资源，导致后续所有正常用户的请求被强行滞留在等待队列（Waiting Queue）中。宏观上表现为系统的首 Token 延迟（TTFT）呈指数级飙升，直至服务彻底瘫痪。

	\subsection{缓存侧信道（Side-Channel）攻击面：内存墙穿透}
	针对 \ref{sec: 高性能推理系统的内存调度机制} 节中 Prefix Caching 的共享复用机制，攻击者可实施更为隐蔽的缓存污染（Cache Pollution）侧信道攻击。

	在多租户环境中，正常用户的请求通常符合长尾的 Zipf 分布。攻击者为了破坏这种局部性，故意构造大量前缀互不相同、符合均匀分布（Uniform Distribution）的请求。这些“冷请求”进入系统后，由于无法命中任何哈希树节点，迫使 PagedAttention 机制不断在物理显存中分配新的物理块。当 80GB 的 A100 显存池被这些无意义的冷请求填满后，系统被迫触发 LRU（最近最少使用）淘汰机制，将正常用户的“热点缓存块”无差别驱逐。
	
	这种攻击的致命之处在于：它并没有耗尽网络连接，也没有发送极高并发，但它通过逻辑机制迫使正常用户的后续请求发生 Cache Miss。正常请求不得不从轻量级的解码（Decode）阶段退回到沉重的预填充（Prefill）阶段重新计算。这不仅极大地浪费了显存带宽（引发“内存墙”阻塞），还导致系统吞吐量断崖式下跌。

	\section{异常检测与机器学习分类理论}
	
	面对上述隐蔽且复杂的底层资源压榨攻击，传统的固定阈值告警机制显得捉襟见肘。本文引入滑动窗口机制与 LightGBM 机器学习算法，构建了从特征工程到多分类识别的完整理论链路。

	\subsection{时间序列统计特征与双峰判定}
	单个请求的延迟往往包含系统随机噪声，为了过滤瞬态抖动，本文引入了滑动时间窗口（Sliding Window）机制。给定按时间戳排序的观测序列 $\mathcal{X} = \{x_1, x_2, \dots, x_N\}$，对于大小为 $W$ 的滑动窗口，当前时刻 $t$ 的局部均值 $\mu_t$ 与局部标准差 $\sigma_t$ 分别定义为：
	\begin{equation}
		\mu_t = \frac{1}{W} \sum_{i=t-W+1}^{t} x_i, \quad \sigma_t = \sqrt{\frac{1}{W-1} \sum_{i=t-W+1}^{t} (x_i - \mu_t)^2}
	\end{equation}

	此外，本文在实验中发现，正常流量与攻击流量在 TTFT 分布上会呈现出显著的“双峰分离（Twin Peaks）”现象。为了从数学上刻画这种分布畸变，引入了峰度（Kurtosis）这一高阶统计量：
	\begin{equation}
		\text{Kurtosis}(\mathcal{X}) = \frac{\mathbb{E}[(\mathcal{X} - \mu)^4]}{\sigma^4} - 3
	\end{equation}
	在遭受缓存侧信道攻击时，TTFT 的分布将显著右移并拉出长尾，表现为峰度指标的剧烈突变。本文提取的核心多维特征如表\ref{tab: 大模型推理异常检测特征工程清单}所示。

	\subsection{孤立森林（Isolation Forest）与 OOD 检测}
	云端推理系统常面临宿主机网络波动等非攻击造成的偶发性脏数据，这些分布外（OOD）异常点会干扰监督模型的训练。本文引入孤立森林（Isolation Forest）算法进行前置数据清洗。
	孤立森林的核心假设是异常样本在特征空间中具有“少而不同”的特性。对于输入样本 $x$，其异常评分 $s(x, n)$ 定义为：
	\begin{equation}
		s(x, n) = 2^{-\frac{\mathbb{E}(h(x))}{c(n)}}
	\end{equation}
	其中：$h(x)$ 为样本 $x$ 在随机树中的路径长度；$\mathbb{E}(h(x))$ 是 $x$ 在多棵树中路径长度的期望值；$c(n)$ 是包含 $n$ 个样本的二叉搜索树的平均路径长度。当 $s$ 接近 1 时，即可判定该样本为系统噪声并予以隔离。

	\subsection{基于 LightGBM 的多维特征分类算法}
	在完成特征提取与异常点清洗后，攻击检测实质上转化为一个多分类问题（Label 0: 正常, Label 1: DDoS, Label 2: 侧信道）。本文采用轻量级梯度提升机（LightGBM）作为核心分类器，其基于直方图（Histogram-based）和单边梯度采样（GOSS）的决策树算法，不仅精度高，且推理延迟极低，避免了防御模块反向拖垮大模型推理性能。完整的特征提取与检测流程如算法\ref{alg: 特征提取与检测}所示。

	\begin{algorithm}[!h]
		\caption{基于滑动窗口的大模型推理异常检测算法} \label{alg: 特征提取与检测}
		\Input{1) 实时推理日志数据流 $\mathcal{S}$; 2) 滑动窗口大小 $W$; 3) 异常阈值 $\tau$。}
		\Output{当前系统状态标签（Normal, DDoS, Side-Channel）。}
		初始化特征缓冲区 $\mathcal{B} = \emptyset$\;
		
		\For(\tcc*[f]{持续监听 vLLM 日志}){新请求记录 $r_t \in \mathcal{S}$ 到来}{
			提取 $r_t$ 的 TTFT, TPOT, 命中率等基础指标并压入 $\mathcal{B}$\;
			
			\eIf(\tcc*[f]{缓冲区已满，开始计算}){当前缓冲区大小 $|\mathcal{B}| \geq W$}{
				计算窗口内的滑动统计量：$\mathbf{F}_t = [\mu_{t}, \sigma_{t}, \text{Kurtosis}_{t}, \dots]$\;
				利用孤立森林模型计算异常得分 $s$\;
				
				\eIf(\tcc*[f]{OOD 数据拦截}){$s > \tau$}{
					触发系统未知异常告警，丢弃脏数据样本\;
				}{
					\tcp{送入 LightGBM 分类器进行攻击定性}
					输入 $\mathbf{F}_t$ 至预训练的 LightGBM 模型\;
					得出预测标签 $L_t \in \{0, 1, 2\}$\;
					\textbf{return} $L_t$\;
				}
				移除缓冲区中最旧的数据，维持窗口滑动\;
			}{
				继续收集数据，保持系统静默\;
			}
		}
	\end{algorithm}

	\subsection{机器学习分类评价指标与超参数优化理论}
	
	在构建异常检测模型时，科学的评估体系与参数寻优算法是不可或缺的理论基础。

	\textbf{1. 分类任务评价指标} \\
	针对大模型攻击检测任务，本文采用混淆矩阵（Confusion Matrix）衍生的标准评价指标。设 $TP$ 为真正例（预测为攻击且实际为攻击），$TN$ 为真负例（预测为正常且实际为正常），$FP$ 为假正例（误报），$FN$ 为假负例（漏报）。各项指标的数学定义如下：
	\begin{itemize}
		\item \textbf{准确率（Accuracy）}：模型整体预测正确的样本比例。
		\begin{equation}
			Accuracy = \frac{TP + TN}{TP + TN + FP + FN}
		\end{equation}
		\item \textbf{精确率（Precision）}：衡量模型报警的准确性，用于评估误报率。
		\begin{equation}
			Precision = \frac{TP}{TP + FP}
		\end{equation}
		\item \textbf{召回率（Recall）}：衡量模型抓取真实攻击的能力，对于安全防护系统尤为重要，旨在降低漏报率。
		\begin{equation}
			Recall = \frac{TP}{TP + FN}
		\end{equation}
		\item \textbf{F1-Score}：精确率与召回率的调和平均数，用于在正负样本比例不平衡（如攻击样本远少于正常样本）时，综合评估模型性能。
		\begin{equation}
			F1 = 2 \times \frac{Precision \times Recall}{Precision + Recall}
		\end{equation}
	\end{itemize}

	\textbf{2. 贝叶斯优化（Bayesian Optimization）} \\
	LightGBM 等树模型包含众多超参数。传统的网格搜索（Grid Search）极耗算力，而本文引入的贝叶斯优化理论，通过构建目标函数的后验概率分布（通常采用高斯过程 Gaussian Process），并最大化采集函数（Acquisition Function），能够以极少的迭代次数在全局参数空间中逼近最优解，这为后续章节中的轻量级检测模型调优提供了算法支撑。

\section{算力枯竭型攻击场景与检测系统架构}

	传统的网络层 DDoS 攻击旨在耗尽目标服务器的带宽或 TCP 连接数 \cite{针对分布式拒绝服务泛洪攻击的防御机制综述}，而针对大模型的攻击则表现出“网络流量合法、底层算力透支”的隐蔽特征。这在本质上属于一种针对神经网络硬件消耗的能量-延迟攻击（Energy-Latency Attacks），其机理类似于 Shumailov 等人提出的“海绵样本（Sponge Examples）”概念 \cite{海绵样本针对神经网络的能量与延迟攻击}。攻击者通过构造极其耗费硬件算力的特定载荷，以极小的网络开销瞬间打满 GPU 的运算上限。本节将从底层浮点运算量与排队论的角度，明确该攻击场景的物理机理，并提出相应的检测系统整体架构设计。

	\subsection{算力压榨攻击的物理机理与数学建模}
	根据第二章对 Transformer 架构的理论分析，大模型的单次推理物理上被割裂为计算密集型的预填充（Prefill）阶段与访存密集型的解码（Decode）阶段。对于输入长度为 $L$ 的提示词序列，其在预填充阶段主要经历线性映射、注意力机制（Attention）与前馈神经网络（FFN）的计算。单次预填充的浮点运算量（FLOPs）可近似建模为：
	\begin{equation}
		\text{FLOPs}_{prefill}(L) \approx 24 \cdot B \cdot L \cdot d_{model}^2 + 4 \cdot B \cdot L^2 \cdot d_{model}
	\end{equation}
	其中，$B$ 为当前批处理大小（Batch Size），$d_{model}$ 为模型的隐藏层维度。可以看出，预填充阶段对 GPU 内部的 Tensor Core 算力占用极高，是典型的计算墙（Compute-bound）任务。

	在本文定义的威胁模型中，攻击者拥有合法的 API 访问权限。为了最大化攻击效率，攻击者往往倾向于采用“短文本脉冲”策略。本文引入网络-算力放大因子（Amplification Factor, $\eta$）来衡量攻击的破坏力，其定义为请求触发的底层算力消耗与其在网络传输中占用的字节开销之比：
	\begin{equation}
		\eta = \frac{\text{FLOPs}_{prefill}(L)}{\text{Size}_{bytes}(L)}
	\end{equation}
	由于自然语言文本的网络体积（通常仅为几百 Bytes）极小，而引发的 FLOPs 却高达 $10^{12}$（TFLOPs）级别，这种不对称性赋予了攻击者数亿倍的放大因子。极高的 $\eta$ 值使得攻击流量能够无损穿透基于带宽限流的传统 Web 应用防火墙（WAF）。

	当攻击者在极短时间内发送并发度为 $CONC$ 的海量短文本请求时，GPU 的理论峰值算力 $C_{GPU}$ 会被瞬间打满。依据排队论模型，一旦请求的到达率（Arrival Rate, $\lambda$）远大于系统的服务率（Service Rate, $\mu$），系统调度器将被迫把无法即时处理的请求置入等待队列（Waiting Queue）。此时，第 $k$ 个正常用户请求的首 Token 延迟（$TTFT_k$）将不可避免地承受前方恶意请求带来的级联排队惩罚：
	\begin{equation}
		TTFT_k = T_{compute}^{(k)} + \sum_{i=1}^{Q_{waiting}} \frac{\text{FLOPs}_{prefill}(L_i)}{C_{GPU}}
	\end{equation}
	式中，累加项代表了在队列中等待前方所有任务计算完成的阻塞耗时。由于 DDoS 攻击导致 $Q_{waiting}$ 呈陡峭斜率增长，该阻塞耗时将远大于正常的 $T_{compute}^{(k)}$，从而在宏观上表现为应用层延迟的指数级飙升与系统可用性的彻底瘫痪。

	针对上述应用层语义合法、底层资源异常的威胁场景，并受限于真实生产环境的物理隔离与合规限制，本文搭建了基于受控物理机的仿真与检测一体化架构。如图\ref{fig: 异常检测系统总体架构}所示，该架构实现了负载压测、核心推理与旁路检测的模块化解耦。

	\begin{figure}[!htb]
		\centering
		\includegraphics[width=0.95\linewidth]{dc_系统总体架构} 
		\caption{面向大语言模型推理服务的异常检测仿真系统总体架构} \label{fig: 异常检测系统总体架构}
	\end{figure}

	该总体架构根据数据流向与功能边界，划分为四个核心模块：
	\begin{enumerate}
		\item \textbf{流量仿真层}：作为整个受控环境的流量源，替代了真实业务中的物理网关。该层依托异步协程并发框架，读取经过预处理与分档的开源真实对话数据集，通过信号量（Semaphore）严格控制并发度（CONC）与发包的概率分布（如 Zipf 或 Uniform 分布）。其核心职责是向目标系统定向注入平稳的长尾基线流量，或模拟极具破坏力的脉冲式 DDoS 与侧信道攻击载荷。
		\item \textbf{推理服务器}：作为系统的核心算力底座，承载大语言模型的物理计算过程。引擎内部维护着连续批处理（Continuous Batching）调度器与 PagedAttention 显存物理块。该层不仅是处理合法推理请求的执行域，也是攻击者压榨 Tensor Core 算力与干扰物理缓存的受灾靶点。
		\item \textbf{全链路监控层}：采用独立于主计算链路的旁路（Sidecar）模式部署，以避免对目标引擎引入额外的观测开销（Observer Effect）。该层通过高频轮询机制与底层 API，实时监听并抓取应用层的端到端时延（TTFT 与 TPOT），以及引擎内部深度的物理队列状态（如 Waiting 和 Running 队列长度）。
		\item \textbf{分析与检测层}：作为整个防御体系的检测核心，接收采集层输出并经过时间戳对齐的高维时序特征矩阵。利用本文优化构建的轻量级多分类模型（LightGBM），对当前滑动窗口内的系统流量状态进行特征解构与模式识别，最终输出当前状态为正常业务、算力枯竭攻击或缓存污染攻击的决策标签。
	\end{enumerate}

\section{请求生命周期分析与异常特征溯源}
\label{sec: 请求生命周期分析与异常特征溯源}
	为了从底层逻辑上解释算力枯竭引发的系统宏观畸变，并为后续的多维特征提取提供理论合法性，本节深入剖析了大模型推理请求的微观执行链路及跨层状态映射规律。

	\subsection{推理请求的微观执行链路与延迟解构}

	传统的静态批处理（Static Batching）在处理大语言模型变长输出时，极易产生严重的显存碎片与算力闲置。为了突破这一瓶颈，Orca 率先提出了迭代级调度（Iteration-level Scheduling）与连续批处理（Continuous Batching）机制 \cite{Orca基于Transformer的生成模型的分布式服务系统}。该机制允许新到达的请求在当前批次完成单次迭代后立即动态插入，从而极大地提升了系统整体的并发吞吐量。本文研究的 vLLM 底层调度器正是继承并高度优化了这一经典架构。

	在大语言模型推理服务中，单个请求从客户端发起到最终接收响应的完整生命周期包含多个复杂的物理与逻辑流转节点。如图\ref{fig: 大模型请求生命周期}所示，本文将该执行链路精细解构为网络接入、排队与调度、预填充以及解码生成等核心阶段，并明确了各阶段对总延迟（Total Latency）的物理贡献。

	\begin{figure}[!htb]
		\centering
		\includegraphics[width=0.95\linewidth]{dc_请求的生命周期} 
		\caption{大语言模型推理请求微观执行链路与延迟构成分析} \label{fig: 大模型请求生命周期}
	\end{figure}

	\begin{enumerate}
		\item \textbf{客户端请求与网络接入（Client Request \& Network Access）} \\
		推理生命周期始于客户端发起请求。请求数据经由广域网传输至云端的 API 网关或负载均衡器（Load Balancer）。此阶段产生的基础耗时受限于物理网络带宽与路由延迟，属于系统外部的网络传输开销。

		\item \textbf{阶段一：排队与调度（Queuing \& Scheduling）} \\
		请求到达引擎后，首先被置入全局请求队列。系统的调度器（Scheduler）基于当前并发请求量、系统设定的最大批处理容量（Max Batch Size）以及当前可用物理显存大小（VRAM Size）进行综合调度决策。由于显存大小直接决定了系统可容纳的 KVC（Key-Value Cache）缓存上限，KVC 管理效率在此间接影响了请求的准入率。滞留在该阶段的请求将产生\textbf{排队时延（Queuing Latency）}。

		\item \textbf{阶段二：预填充计算（Prefill Phase）} \\
		调度成功的请求批次进入预填充阶段。该阶段充分调用 GPU 执行海量并行计算，集中处理所有输入提示词（提示词长度直接正相关于计算量），并计算出每个 Token 对应的 KVC 以为后续生成做准备。此阶段产生\textbf{并行计算时延}。当该阶段结束并输出首个结果时，即标志着关键指标\textbf{首字时延（TTFT, Time to First Token）}的产生。

		\item \textbf{阶段三：解码与生成（Decode \& Generation）} \\
		首字输出后，请求状态转入逐个 Token 生成的串行解码循环。系统在每轮生成后必须动态更新 KVC。此阶段产生\textbf{串行计算时延}，其生成速率受制于目标输出长度、模型参数规模、GPU 硬件计算能力以及显存带宽。系统将不断循环“判断是否完成生成”的逻辑，直至遇到停止符或达到最大长度。

		\item \textbf{网络传输与生命周期终止} \\
		生成逻辑结束后，最终的序列结果经由网络再次传输返回客户端，客户端接收完毕即标志着该请求生命周期的彻底终结。
	\end{enumerate}

	综上物理流转过程，大模型单次推理的总延迟宏观上可严格解构为：
	\begin{equation}
		T_{total} = T_{network\_rtt} + T_{queueing} + T_{compute\_parallel}^{(prefill)} + T_{compute\_serial}^{(decode)}
	\end{equation}
	式中，$T_{network\_rtt}$ 为网络往返传输耗时。在本文所关注的算力枯竭型 DDoS 攻击场景下，攻击者正是通过极大地推高“阶段二”的并行计算负载，迫使整个系统调度阻塞，进而导致后续正常请求的 $T_{queueing}$ 发生指数级膨胀，最终诱发服务瘫痪。

\subsection{物理资源瓶颈的跨层状态映射与级联失效}
	在上述生命周期中，硬件层（Hardware Layer）的算力或显存瓶颈并非孤立存在，而是会通过系统调度层（Scheduler Layer）不可逆地向应用层（Application Layer）传播，从而引发可被外部旁路观测的多维时序畸变。这种自下而上的“跨层状态映射”是实现高精度异常检测的物理前提。

	当高并发短文本 DDoS 攻击发生时，系统状态的映射严格遵循以下“级联失效（Cascading Failure）”演进逻辑：
	
	\begin{enumerate}
		\item \textbf{底层物理资源透支（算力层畸变）}：攻击者注入的高频短文本脉冲瞬间抵达引擎，在生命周期的“阶段二（预填充计算）”触发海量的并行矩阵乘法任务。这导致 GPU 的 Tensor Core 算力被瞬间极度透支，浮点运算单元处于持续满载的阻塞状态。
		\item \textbf{中层调度队列堆积（系统层畸变）}：由于底层算力枯竭，正在执行的批处理任务（Batch）耗时被严重拉长，vLLM 引擎的连续批处理（Continuous Batching）调度器无法及时释放计算槽位，进而无法将新的请求从 Waiting Queue 推进至 Running Queue。系统吞吐量（Throughput）发生断崖式下跌，物理层的阻塞直接具象化为调度队列的宏观堆积。此时，Waiting 队列长度 $Q_{waiting}$ 呈现出异常的陡峭斜率（即其时间导数 $dQ_{waiting}/dt \gg 0$）。
		\item \textbf{顶层端到端时延崩塌（应用层畸变）}：系统级的排队阻塞最终无可避免地向应用层传导。客户端观测到的不仅是平均时延的上升，更是首字时延（TTFT）极度严重的“长尾效应（Heavy Tail）”。正如 Google 学者 Dean 等人 \cite{规模化环境下的长尾延迟问题} 所指出的，规模化分布式系统中的长尾延迟会成倍放大组件级的性能波动。在 DDoS 脉冲下，正常状态下毫秒级的 TTFT 会呈指数级飙升至数十秒，直接导致服务级别协议（SLA）的全面违约。
	\end{enumerate}

	\textbf{特征降维与多维交叉监控的必要性}：
	单独观测任何单一维度的指标，均无法实现对大模型 DDoS 攻击的精准归因。例如，纯应用层的 TTFT 飙升可能是由突发的合法超长文本（如 12k Context）或偶发网络拥塞引起；而纯底层的 GPU 高负载本就是模型推理的常态。因此，本文揭示的这种从“物理算力透支”到“系统队列膨胀”，最终表现为“应用层时延长尾畸变”的跨层映射规律，在理论上证明了：必须摒弃单一时延或单一网络特征，转而构建深度融合应用层、系统层与硬件层状态的\textbf{多维交叉特征工程（Multi-dimensional Cross-layer Feature Engineering）}，才能在复杂的云原生背景流量中精准剥离出隐蔽的资源压榨型攻击。

\section{压测负载制备与多维特征工程构建}

	前文的跨层映射分析表明，大模型 DDoS 攻击的本质是特定负载在物理算力上的投影畸变。为了科学评估防御模型的性能，本文必须首先构建贴近真实场景的压测负载（Workload），继而在受控环境中将其转化为可供机器学习算法摄入的高维时序特征矩阵（Feature Matrix）。本节将详细阐述这一从“非结构化文本”到“结构化特征向量”的端到端制备流水线。

	\subsection{基于 ShareGPT 的压测负载数据集制备}
	在真实的 MaaS 云服务中，用户输入的提示词（Prompt）长度往往呈现出极大的方差。若仅采用人工合成的随机定长文本进行测试，极易导致防御模型对特定长度过拟合。为此，本文选取了开源社区广泛使用的大规模真实对话数据集 ShareGPT 作为基础数据池，并实施了严格的清洗与分档重构策略。

	\textbf{1. 数据清洗与规范化分词} \\
	针对 ShareGPT 原始的 JSON 结构，本文提取每一轮对话中代表人类指令的 \texttt{human} 字段文本。为了精准度量文本对 GPU 算力的真实消耗，本文调用了目标推理模型（如 Qwen-1.5）官方配套的字节对编码分词器（BPE Tokenizer），将离散文本强制映射为 Token 序列，并统计其确切长度 $L_{prompt}$。清洗流水线遵循以下过滤准则：
	\begin{itemize}
		\item 剔除包含大量非标准 Unicode 乱码或纯 HTML 标签的失效请求。
		\item 剔除 $L_{prompt} < 5$ 的极短文本（此类文本通常为无意义的标点或助词，不具备真实业务价值）。
		\item 剔除 $L_{prompt} > 16384$ 的超长文本（此类文本极易直接触发 vLLM 引擎的显存溢出 OOM 崩溃，从而掩盖需要观测的排队堆积特征）。
	\end{itemize}

	\textbf{2. 攻击载荷与正常基线的数据分档}

	\begin{enumerate}
		\item \textbf{短文本压榨集（Short-1k）}：筛选 $L_{prompt} \le 750$ 的数据构成（平均长度约 42.2 Tokens）。该数据集在实验中作为恶意攻击载荷，配合极高的并发度（如 $CONC=400$）定向注入引擎，其极短的序列特征使其能够在几乎不占用额外显存池的情况下，瞬间打满 A100 GPU 的 Tensor Core 浮点运算上限。
		\item \textbf{长文本基线集（Long-12k）}：筛选 $L_{prompt} \in [900, 6000]$ 的数据构成（最高达 5781 Tokens，平均约 2072.3 Tokens）。该数据集具备极强的长尾上下文特征，在实验中配合低并发度持续发送，用于精准模拟云端平稳且消耗大量 KV Cache 的正常业务背景流量。

	单一时间窗口极易受到网络波动的干扰。正如 Feinstein 等人在传统分布式拒绝服务攻击研究中所指出的，高阶统计学方法在检测高并发泛洪攻击时具有极高的敏锐度 \cite{分布式拒绝服务攻击的统计学检测与响应方法}。受此启发，本文进一步引入宏观与微观的多时间尺度（Multi-Scale）分析，计算时延概率分布的香农信息熵与峰度。对于时刻 $t$ 的系统观测状态，本文定义一个包含 $N$ 个近期请求的滑动观测窗口 $\mathcal{W}_t = \{r_{t-N+1}, r_{t-N+2}, \dots, r_t\}$。在该窗口内，探针模块持续提取并计算以下三类核心特征：

	\textbf{1. 应用层表现特征（Application-level Features）} \\
	该维度直接映射客户端的服务体验，包含首字时延（TTFT）与每 Token 平均输出时延（TPOT）。对于窗口内的任意请求 $r_i$，其基础时延定义为：
	\begin{equation}
		TTFT_i = T_{first\_token}^{(i)} - T_{arrive}^{(i)}
	\end{equation}
	在此基础上，提取窗口 $\mathcal{W}_t$ 内 TTFT 的均值 $\mu_{TTFT}$、方差 $\sigma_{TTFT}^2$ 以及 95 分位点延迟（$P95_{TTFT}$）。针对 DDoS 攻击下的突变特性，特别引入一阶差分最大值 $\max(\Delta TTFT)$ 以捕捉微观脉冲。

	\textbf{2. 系统级调度特征（System-level Scheduling Features）} \\
	该维度旨在穿透应用层语义，直接监测 vLLM 引擎底层的物理资源调度瓶颈。旁路探针通过引擎开放的 \texttt{/metrics} 端点，以 50 毫秒为固定周期异步抓取运行队列（Running Queue）与等待队列（Waiting Queue）的长度。为了量化队列的“堆积剧烈程度”，本文计算队列长度的时间导数（即增长斜率）：
	\begin{equation}
		Slope(Q_{waiting}) = \frac{dQ_{waiting}}{dt} \approx \frac{Q_{waiting}(t) - Q_{waiting}(t-\Delta t)}{\Delta t}
	\end{equation}
	在算力透支场景下，该斜率特征将表现出对异常流量极其敏锐的阶跃响应。

	\textbf{3. 时间序列高阶统计特征（High-order Statistical Features）} \\
	单一时间窗口极易受到网络波动的干扰。本文进一步引入宏观与微观的多时间尺度（Multi-Scale）分析。对于由窗口内序列构成的时延概率分布 $P(x)$，本文计算其香农信息熵（Shannon Entropy）以衡量系统的混乱度：
	\begin{equation} \label{eq: 特征信息熵}
		H(\mathcal{W}_t) = - \sum_{x \in \mathcal{W}_t} P(x) \log_2 P(x)
	\end{equation}
	此外，为了精准捕捉排队效应导致的严重“长尾”现象，特征库中额外融合了四阶中心矩——峰度（Kurtosis，$K$）：
	\begin{equation}
		K(\mathcal{W}_t) = \frac{\frac{1}{N} \sum_{i=1}^{N} (TTFT_i - \mu_{TTFT})^4}{\sigma_{TTFT}^4}
	\end{equation}
	当系统遭受攻击导致长尾畸变时，峰度值将显著偏离正态分布的基准值（$K=3$）。

	综上，经过数据清洗、旁路采集与多尺度高阶统计计算，每个时间步 $t$ 下的非结构化流量负载，最终被严格映射为一个 18 维的连续特征向量 $\mathbf{F}_t \in \mathbb{R}^{18}$。该特征矩阵不仅包含了表观的时延恶化，更深度融合了底层队列堆积与分布畸变的物理参量，为后续 LightGBM 分类器在高维空间中绘制决策边界提供了极高的数据保真度。

\section{异常检测分类器设计与超参数寻优}

	在前述高维交叉特征工程（包含 18 维表观与底层特征）构建完成后，亟需引入一种低延迟、高精度的机器学习分类器，以在复杂的云端背景流量中划定正常的业务边界。本节将详细阐述分类模型的选型逻辑、内部核心机制以及基于先验的超参数空间搜索策略。

	\subsection{基于贝叶斯优化的空间搜索策略}
	在构建树模型时，超参数的选取直接决定了模型在过拟合与欠拟合之间的平衡状态。由于大模型防御对“检测实时性”要求极高，传统的网格搜索（Grid Search）耗时过长，且容易陷入局部最优。因此，本文引入了基于高斯过程（Gaussian Process, GP）的实用贝叶斯优化（Bayesian Optimization）框架 \cite{机器学习算法的实用贝叶斯优化}。

	设定模型的超参数空间为 $\mathcal{X}$，待优化的黑盒目标函数（本文定义为验证集上的 F1-Score）为 $f(x)$。贝叶斯优化的核心在于通过已有的观测集 $\mathcal{D}_{1:t} = \{(x_1, y_1), \dots, (x_t, y_t)\}$，更新目标函数的后验概率分布，并通过最大化采集函数（Acquisition Function，本文采用期望提升 Expected Improvement, EI）来决定下一组采样的超参数 $x_{t+1}$：
	\begin{equation}
		EI(x) = \mathbb{E} \left[ \max(0, f(x) - f(x^+)) \mid \mathcal{D}_{1:t} \right]
	\end{equation}
	其中 $x^+$ 为当前已观测到的最优超参数组合。通过该策略，本文在极小的算力开销下，完成了对决策树叶子节点数（\texttt{num\_leaves}）、学习率（\texttt{learning\_rate}）等核心参数的全局寻优，为防御模型的轻量化部署奠定了参数基础。

	\subsection{面向大模型防御的 LightGBM 树模型构建}
	在具体算法选型上，本文摒弃了计算开销极大的深度神经网络（DNN），转而采用基于直方图优化的轻量级梯度提升树模型（LightGBM）\cite{LightGBM高效梯度提升决策树}。除了极低的推理延迟外，其内置的单边梯度采样（GOSS）机制与大模型攻击场景具有天然的物理契合度。

	在真实的 MaaS 系统中，正常的长尾流量（Label 0）占据绝对多数，而高并发的 DDoS 攻击脉冲（Label 1）属于极少数的异常点，导致训练样本极度不平衡。GOSS 机制在计算信息增益时，主动保留产生巨大误差的攻击样本（即拥有大梯度的样本点），而对海量的正常平稳样本（小梯度样本点）进行随机下采样，并引入常数乘子 $\frac{1-a}{b}$ 以补偿由于下采样造成的数据分布偏差：
	\begin{equation}
		\tilde{V}_j(d) = \frac{1}{n} \left( \frac{\left( \sum_{x_i \in A_l} g_i + \frac{1-a}{b} \sum_{x_i \in B_l} g_i \right)^2}{n_l^j(d)} + \frac{\left( \sum_{x_i \in A_r} g_i + \frac{1-a}{b} \sum_{x_i \in B_r} g_i \right)^2}{n_r^j(d)} \right)
	\end{equation}
	式中，$A$ 为大梯度样本子集，$B$ 为小梯度样本子集。该机制确保了分类器在面对瞬息万变的大模型队列畸变时，能够快速收敛并对少数派的攻击特征保持极高的敏感度。


	为了更加清晰地展示本文检测模型的执行逻辑，算法\ref{algo: LightGBM检测算法}给出了融合贝叶斯寻优与 GOSS 采样机制的端到端训练伪代码。该算法流严格输入前文提取的 18 维高阶时序特征，并在极度不平衡的攻击样本下实现决策树的快速迭代与收敛。

	\begin{algorithm}[H]
		\caption{基于多尺度特征与 GOSS 机制的 LightGBM 检测算法}
		\label{algo: LightGBM检测算法}
		\KwIn{训练特征矩阵 $\mathcal{D} = \{(\mathbf{F}_i, y_i)\}_{i=1}^N$；贝叶斯寻优最大迭代次数 $T_{max}$；大梯度样本保留比例 $a$；小梯度样本采样比例 $b$；}
		\KwOut{最优异常检测模型 $\mathcal{M}^*$}
		
		\tcp{第一阶段：全局超参数空间寻优}
		初始化贝叶斯高斯过程（Gaussian Process）先验分布 $\mathcal{GP}$\;
		\For{$t \leftarrow 1$ \KwTo $T_{max}$}{
			基于当前 $\mathcal{GP}$，通过最大化期望提升（EI）采集函数获取当前轮超参数集 $\theta_t$\;
			
			\tcp{第二阶段：基于 GOSS 的轻量级树模型训练}
			初始化当前超参数下的模型 $\mathcal{M}_{\theta_t}^{(0)}$\;
			\For{$k \leftarrow 1$ \KwTo $K$ \text{ (单模型最大树深或迭代数)}}{
				计算集合中所有样本的一阶梯度 $g_i$ 与二阶导数 $h_i$\;
				根据 $|g_i|$ 对样本进行降序排列\;
				选取前 $a \times N$ 个样本构成大梯度集合 $A$ \tcp*{保留核心攻击样本}
				从剩余的 $(1-a) \times N$ 个样本中随机无放回采样 $b \times N$ 个样本构成集合 $B$\;
				\For{每个样本 $x_i \in B$}{
					$g_i \leftarrow g_i \times \frac{1-a}{b}$ \tcp*{为小梯度样本引入常数补偿权重}
				}
				基于集合 $A \cup B$ 生成轻量级特征直方图（Histogram）\;
				在直方图上寻找最优特征分裂点，生成新决策树 $T_k$\;
				更新当前模型预测值：$\mathcal{M}_{\theta_t}^{(k)} = \mathcal{M}_{\theta_t}^{(k-1)} + \eta \cdot T_k$\;
			}
			在交叉验证集上计算模型 $\mathcal{M}_{\theta_t}$ 的 F1-Score，记为 $f(\theta_t)$\;
			将观测对 $(\theta_t, f(\theta_t))$ 存入历史集，并更新 $\mathcal{GP}$ 的后验概率分布\;
		}
		获取具有最大验证集 F1-Score 的超参数 $\theta^*$ 及其对应的最终模型 $\mathcal{M}^*$\;
		\Return $\mathcal{M}^*$\;
	\end{algorithm}

\section{实验评估与检测性能分析}

	本节将在受控的仿真环境中，对上述多维特征工程与 LightGBM 检测架构的有效性进行全面评估，并通过对比实验与特征消融，验证本文所提防御体系的理论正确性与工程鲁棒性。

	\subsection{实验环境部署与目标模型选型}
	为保证压测现象的客观性与实验结果的工业界参考价值，本文在受控的物理节点上完成了防御框架的端到端部署。物理计算节点搭载单张 NVIDIA A100 Tensor Core GPU (80GB HBM2e) 高性能加速卡，操作系统为 Ubuntu 22.04 LTS，核心推理服务框架采用高度优化的 vLLM 引擎。

	\textbf{1. 目标推理模型选型与基准确立} \\
	在评估大模型底层资源压榨攻击时，目标靶机模型的网络结构与参数规模直接决定了系统的计算基准。本文选取了当前开源社区极其先进且广泛应用的 \texttt{Qwen2.5-7B-Instruct} 作为核心目标模型（同时在前期探测中以 \texttt{Qwen1.5-1.8B} 作为微型对照组进行基础验证）。
	选用 \texttt{Qwen2.5-7B-Instruct} 的核心工程考量在于：其采用了最新的因果解码器（Causal Decoder-only）架构，全面支持分组查询注意力机制（GQA）与超长上下文（Long Context）。这使其在并行处理大规模短文本或长尾正常业务时，能够极其高效地触发 vLLM 底层的 PagedAttention 显存分页机制，高度契合本文所定义的多维底层特征（如 Waiting 队列斜率）的探测需求。

	\textbf{2. 受控环境显存资源推演（Resource Deduction）} \\
	为了在单卡物理机上精准隔离并放大算力枯竭与队列阻塞现象，本文对 vLLM 引擎的显存分配上限进行了严格的“受控降维”。
	在默认的半精度浮点数（BF16）格式下，7B 参数量规模的模型静态权重（Weights）约固定占用 14GB 的显存。在本实验中，本文刻意将引擎的显存分配比例（\texttt{gpu\_memory\_utilization}）严格限制在 0.3 至 0.4 的区间内。
	这意味着，在 A100 提供的 80GB 总容量中，vLLM 仅被授权使用约 24GB 到 32GB 的物理显存。扣除 14GB 的权重占用与少量的 CUDA 上下文开销后，引擎实际可用于 PagedAttention 动态调度的物理 KV Cache 块（Blocks）空间被精准锁定在 10GB 至 18GB 之间。
	这种“先进模型、受控显存池”的边界约束极其巧妙：它成功构建了一个敏感度极高的脆弱算力底座。当注入并发度为 $CONC=400$ 的短文本脉冲时，系统将迅速触及 Tensor Core 算力与 KV Cache 的双重物理阈值，确保探针能够捕获到最纯净、最剧烈的级联失效（Cascading Failure）特征，而不会因 A100 原始的 80GB 庞大显存掩盖了早期的微观畸变。

	\textbf{3. 正常业务基线状态标定（Baseline Calibration）} \\

	\section{缓存侧信道攻击的物理映射与威胁机理}
	
	如第二章所述，现代大模型推理框架广泛采用了基于 PagedAttention 的前缀缓存技术。这种多租户共享的物理显存池，在极大提升资源利用率的同时，也打破了租户间的物理隔离边界。在微架构安全领域，基于共享内存的缓存侧信道攻击长期以来是破坏多租户云环境物理隔离的核心威胁 \cite{现代硬件上微架构定时攻击与对策的综述}。早期的研究主要利用微架构定时缺陷窃取 AES 等加密算法的密钥 \cite{缓存攻击与对策以AES为例}，如经典的 Flush+Reload 技术 \cite{Flush+Reload高分辨率低噪声的L3缓存侧信道攻击}。随着人工智能的崛起，Yan 等人提出的 CacheTelepathy 证明了攻击者甚至可以通过共享缓存资源逆向推断出云端深度神经网络（DNN）的内部架构参数 \cite{CacheTelepathy利用共享资源攻击学习DNN架构}。而如今，随着 vLLM 等框架广泛采用基于全局物理显存池的 PagedAttention 机制，这种硬件级的威胁被无可避免地引入到了大模型的生成流水线中。

	近年来，针对大语言模型推理阶段的微架构侧信道攻击引起了学术界的高度关注。Song 等人率先揭示了 LLM 服务系统中的定时侧信道漏洞 \cite{早起的鸟儿发现漏洞：揭示大型语言模型服务系统中的定时侧信道}。随后，Carlini 等人证明了在高效推理接口上实施远程定时攻击的可行性 \cite{高效推理接口上的远程定时攻击}。针对大模型独有的生成机制，Zhang 等人利用输出 Token 数量的差异构建了定时侧信道 \cite{输出Token数量的定时侧信道攻击}，而 Zheng 等人则提出了 InputSnatch，通过定时侧信道在共享缓存中窃取用户的私密输入 \cite{共享缓存中的定时侧信道攻击}。

	特别是在大模型推理服务广泛采用前缀缓存（Prefix Caching）技术 \cite{用于高效大型语言模型推理的习得前缀缓存} 以提升并发性能后，Luo 等人深入揭示了多租户共享 KV-Cache 所带来的严重隐私泄露风险 \cite{缓存中的阴影：揭示并缓解KV-缓存在大型语言模型推断中的隐私风险}。此外，McDonald 等人提出了名为 Whisper Leak 的端侧信道攻击方案 \cite{对大型语言模型的端侧信道攻击}，其开源的攻击框架 \cite{对大型语言模型的端侧信道攻击源码} 为本章验证缓存污染机理与设计畸变特征提供了重要的工程参考模型。同时，结合 Patwari 等人提出的独立于硬件的 LLM 推理性能分析模型 \cite{通过硬件无关的分析建模预测LLM推断性能}，本文得以在数学层面更精准地量化缓存命中率对 TTFT 的惩罚代价。

	\subsection{基于命中率的 TTFT 惩罚数学模型}
	在理想状态下，大模型处理携带历史系统提示词（System Prompt）的请求时，其首 Token 延迟（TTFT）的数学期望 $\mathbb{E}[TTFT]$ 很大程度上取决于物理显存块的缓存命中率（Cache Hit Rate, denoted as $P_{hit}$）。
	我们可以将实际的 $TTFT$ 建模为命中与未命中两种状态的加权代价：
	\begin{equation}
		\mathbb{E}[TTFT] = P_{hit} \cdot T_{hit} + (1 - P_{hit}) \cdot T_{miss}
	\end{equation}
	其中，$T_{hit}$ 为命中时的延迟，此时系统仅需执行轻量级的矩阵-向量乘法（GEMV）；$T_{miss}$ 为未命中时的延迟代价（Cache Miss Penalty）。一旦发生未命中，系统必须退回至计算密集型的预填充阶段重新生成 KV Cache，即：
	\begin{equation}
		T_{miss} = T_{GEMM}(L_{prompt}) + T_{wait\_queue} \gg T_{hit}
	\end{equation}
	因此，攻击者只要能人为压低目标系统的 $P_{hit}$，就能在不增加系统整体并发流量的前提下，迫使正常用户的延迟向 $T_{miss}$ 急剧倾斜。

	\subsection{均匀分布流量（Uniform）的缓存污染机制}
	在正常业务模型中，用户的请求内容及其携带的系统提示词（System Prompt）通常服从齐普夫分布（Zipf Distribution）\cite{齐普夫定律与互联网分布}，即少数热点前缀被高频调用。在此分布下，$P_{hit}$ 通常能维持在较高水平。

	然而，缓存侧信道攻击者会故意构造大量前缀完全随机的恶意请求。这些请求在概率统计上严格服从均匀分布（Uniform Distribution）。当这些恶意请求以一定的速率持续注入系统时，它们不仅无法命中已有的哈希树，还会迫使 vLLM 底层的调度器不断为其分配新的物理内存块（Physical Blocks）。当显存池耗尽时，系统触发置换淘汰策略（如 LRU），导致正常用户的“热点缓存块”被无差别驱逐（Eviction）。

	\section{缓存污染实验设计与畸变现象分析}

	为了定量评估上述威胁，本文在 vLLM 压测平台上设计了控制变量实验，通过调节客户端发包的概率分布模型（参数 $\alpha$），精准复现了侧信道干扰下的系统状态。

	\subsection{对照组参数设计与基准测定}
	本实验固定并发度为较低的 $CONC=20$，以排除算力枯竭的干扰。核心变量设定为 Zipf 分布的偏斜度参数 $\alpha$：
	\begin{itemize}
		\item \textbf{正常基线组}：设定 $\alpha=1.2$，模拟具有高度局部复用性的正常用户请求流（Cache Hit 主导）。

	\section{面向侧信道畸变的多维特征解构与检测}

	针对上述物理畸变，单一的均值阈值检测极易与偶尔的网络卡顿混淆。为此，本文依托第三章提出的多尺度特征框架，深入挖掘能够刻画这种分布不对称性的高阶特征参量。

	\subsection{基于高斯混合模型（GMM）的形态学解构}
	对于呈现“双峰”畸变的延迟时间序列，纯粹的均值分析会掩盖真实的系统物理状态。因此，本文在理论上将其近似解构为由隐变量控制的高斯混合模型（Gaussian Mixture Model, GMM）\cite{高斯混合模型理论与应用}：：
	\begin{equation}
		f(x) = w_{hit} \mathcal{N}(x \mid \mu_{hit}, \sigma_{hit}^2) + w_{miss} \mathcal{N}(x \mid \mu_{miss}, \sigma_{miss}^2)
	\end{equation}
	在侧信道攻击的扰动下，$w_{miss}$（即强制重算的惩罚比例）显著上升，引发整体分布峰度（Kurtosis）与偏度（Skewness）的剧变。

	\subsection{微观特征融合与峰度突变监测}
	为了让 LightGBM 敏锐捕捉这种形态学变化，特征矩阵中着重强化了多时间尺度下的峰度特征。实验数据表明，当侧信道攻击启动、缓存命中率发生断崖式下跌时，监测窗口内的 TTFT 峰度值会产生瞬时阶跃。这一高阶统计量构成了本文区分普通网络拥塞与蓄意缓存侧信道攻击的最核心向量。

	\section{本章小结}
	本章深入探索了隐蔽性极强的大语言模型缓存侧信道攻击。通过构建基于命中率的 TTFT 数学惩罚模型，在理论层面揭示了 Uniform 随机流量驱逐 Zipf 热点缓存的内存污染机制。物理压测实验清晰地观测到了攻击流量介入后导致的延迟均值翻倍、长尾拖拽以及经典的“双峰分布”畸变现象，并通过 CDF 曲线量化了服务质量的恶化程度。基于此发现，本文引入了高斯混合模型解构与高阶峰度统计量，进一步完善了多时间尺度特征工程。最后的综合模拟评测结果与系统开销分析证实，本文提出的改进型 LightGBM 检测框架能够精准、低延迟地鉴别多维度的底层资源压榨攻击，为云端大模型 MaaS 服务提供了一套具备高泛化能力的轻量级安全防护方案。
